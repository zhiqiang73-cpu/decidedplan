# Alpha引擎规格书

## 定义

Alpha引擎是一个**信号观察与策略发现机器**。它不持仓、不下单、不交易。它的唯一职责是：

1. 从实时市场数据中发现计算机交易行为的痕迹
2. 验证这些痕迹是否能产出高正确率的入场信号
3. 为每个入场信号找到利润最大化的出场条件
4. 输出完整的策略卡片
5. 管理策略生命周期（上架/下架）

交易执行由独立的执行引擎（execution_engine.py）负责。

## 模块1：实时扫描器 (Scanner)

### 输入
- 每分钟更新的53个特征值

### 工作流程
1. 计算当前所有特征值
2. 检查是否存在**已知的计算机交易痕迹模式**：
   - 该特征组合在历史上触发后，价格方向一致（MFE覆盖率 > 80%）
   - 该模式可归因于物理约束（资金费成本/库存约束/结算规则/执行算法/流量结构）
3. 如果命中 → 发出入场信号给执行引擎
4. 如果发现**未知的特征异常**（从未见过的特征组合进入罕见区域）→ 标记为候选，交给离线验证器

### 不做的事
- 不判断"极端百分位"再看正确率
- 不预设持仓时间
- 不持仓、不下单

## 模块2：离线验证器 (Validator)

### 触发条件
- 扫描器发现新的候选模式
- 定时任务（每天/每周用最新数据重新扫描）
- 手动触发

### 工作流程

#### A. 入场信号验证
1. 取候选特征组合
2. 在历史数据中找到所有触发时刻
3. 计算每次触发后120 bars内的MFE
4. 验证：
   - MFE > 0.04% 的比率 > 80%（方向正确 + 覆盖费用）
   - 样本量 >= 50
   - 可归因于5类物理约束之一
   - Walk-Forward OOS降级 < 15%
   - 触发频率 >= 0.2次/天
5. 全部通过 → 进入出场条件挖掘

#### B. 出场条件挖掘
1. 对每笔历史交易，找到MFE峰值bar
2. 收集所有MFE峰值处的特征值（绝对值 + _vs_entry差值）
3. 用Cohen's d找出MFE峰值的共同特征（d >= 0.2）
4. 构建条件组合：单条件、双条件、三条件
5. 保留Top3，验证"最早触发"策略优于任何单一组合
6. 出场触发条件：当前利润 > 费用 AND 特征命中共同模式

#### C. 止损优化
1. 扫描止损网格：0.10% ~ 1.50%
2. 在每个止损下重新运行出场条件回测
3. 选择PF最高且total_net > 0的止损值

#### D. 输出策略卡片
```
策略名称: [自动生成ID]
方向: LONG / SHORT
物理归因: [约束类型]

入场条件:
  条件1: [特征] [运算符] [阈值]
  条件2: [特征] [运算符] [阈值]

入场验证:
  MFE覆盖率: [百分比]
  样本数: [N]
  触发频率: [次/天]
  OOS降级: [比值]

出场条件 (Top3最早触发):
  C1: [条件]
  C2: [条件]
  C3: [条件]

出场验证:
  PF: [值]
  条件触发时胜率: [百分比]
  总净利润: [百分比]

止损: [百分比]
安全上限: max_hold = 120 bars

物理解释:
  入场: [为什么方向可预测]
  出场: [为什么利润到峰值]
```

## 模块3：策略卡片库 (CardStore)

### 存储
- 文件：`monitor/output/alpha_cards/` 目录
- 每张卡片一个JSON文件
- 包含：入场条件、出场条件、止损、验证指标、物理解释

### 生命周期管理

| 状态 | 含义 | 转换条件 |
|------|------|---------|
| candidate | 扫描器发现，待验证 | 离线验证器开始处理 |
| validating | 正在跑历史验证 | 验证完成 |
| active | 验证通过，执行引擎可使用 | PF下降或手动下架 |
| degraded | PF持续下降，标记观察 | 连续7天PF < 0.8 |
| retired | 已失效，不再使用 | 手动确认或自动 |

### 策略健康监控
- 每天计算每个active策略最近30天的滚动PF
- PF < 1.0 持续3天 → 标记degraded
- PF < 0.8 持续7天 → 自动retired
- 新数据验证PF恢复 > 1.2 → 可重新激活

## 与现有系统的关系

```
现有系统:
  signal_runner.py  →  已有7个策略的入场检测
  smart_exit_policy.py  →  已有策略的出场条件
  execution_engine.py  →  下单执行

Alpha引擎:
  alpha_scanner.py  →  实时扫描（新模块）
  alpha_validator.py  →  离线验证（新模块）
  alpha_card_store.py  →  策略卡片管理（新模块）

集成方式:
  alpha_scanner 发现信号 → 通过与 signal_runner 相同的接口发给 execution_engine
  alpha_validator 产出的策略卡片 → 自动注册到 signal_runner
  已有7个策略 = 手动产出的策略卡片，与Alpha产出的卡片格式完全相同
```

## 关键约束

1. Alpha引擎**不持仓、不下单**，只产出信号和策略卡片
2. 每个发现的信号**必须有物理归因**，纯统计pattern丢弃
3. 入场正确率的定义：**信号是计算机交易痕迹 + 方向正确**，两个同时成立
4. 出场条件：**当前利润覆盖费用 + 特征命中MFE峰值共同模式**
5. 持仓时间**不预设**，是出场条件触发的结果
6. 止损**每个策略独立优化**
7. 策略有生命周期，会失效，需要持续发现新策略替补
